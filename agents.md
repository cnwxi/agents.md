# AGENTS.md

> 适用范围
>
> 本文件定义 AI Coding Agent 在所有仓库中都应遵守的全局工程约束。
>
> 仓库特有的构建、测试、CI、架构、目录结构、依赖和业务规则，应写入对应仓库的 `AGENTS.md`。
>
> 当多个 `AGENTS.md` 同时适用时，应同时遵守全局规则和距离目标文件最近的仓库/子目录规则。

---

# 1. 工作原则

## 必须

- 修改前先理解任务目标。
- 做假设前先搜索现有实现。
- 优先复用已有实现，而不是新增抽象。
- 保持修改最小、可审查、可回滚。
- 保持项目既有代码风格。
- 在可行范围内验证修改。
- 信息不完整时明确说明不确定性。
- 只报告真实执行过的分析、构建、测试和验证结果。

## 禁止

- 猜测 API、接口、配置或运行时行为。
- 修改与任务无关的文件。
- 引入与任务无关的重构。
- 格式化与任务无关的文件。
- 编造分析、测试、构建或验证结果。
- 隐藏失败、警告或未完成事项。

## 应当

- 复用已有工具函数、抽象和项目模式。
- 修改范围严格围绕用户请求。
- 优先采用简单方案，避免过度设计。
- 在可行时保持向后兼容。

---

# 2. 上下文管理

## 必须

- 只读取当前任务所需的文件。
- 优先增量探索，而不是全仓遍历。
- 找到充分证据后立即停止搜索。
- 复用当前任务中已经获得的上下文。
- 优先相信代码、测试和配置中的直接证据。

## 应当

上下文读取优先级：

1. 用户指令
2. 当前文件
3. 直接依赖
4. 配置文件
5. 测试
6. README 或文档
7. 其它源码

## 禁止

- 无明确目的地扫描整个仓库。
- 无新目的地重复读取同一文件。
- 反复总结相同内容。
- 把生成文件或临时笔记当作主要事实来源。
- 将大段日志或大段代码复制到持久化笔记中。

---

# 3. 代码修改

## 必须

- 生成合理范围内最小的 Diff。
- 遵循既有架构和命名习惯。
- 一次只完成一个逻辑修改。
- 除非用户明确要求，否则保持公共行为不变。
- 保持错误处理清晰明确。
- 避免引入不必要的新依赖。

## 应当

- 优先修改已有代码，再考虑新增文件。
- 优先复用已有抽象，再考虑新增抽象。
- 仅在解释非显然意图时添加注释。
- 保持代码易审查、易回滚。

## 禁止

- 重命名无关符号。
- 移动无关文件。
- 引入大范围格式化修改。
- 无故删除有效注释、测试或配置。
- 在最终修改中保留临时调试代码。

# 4. Python 规范

## 必须

- 文件路径处理优先使用 `pathlib`。
- 运行时诊断优先使用 `logging`，不要使用 `print`。
- 公共 API 和非平凡函数应提供类型标注。
- 捕获具体异常，而不是笼统捕获所有异常。
- 保持函数职责单一。

## 应当

- 在适合表达结构化数据时使用 `dataclass`。
- 在表达状态、模式、类型时使用 `Enum`。
- 使用上下文管理器管理资源生命周期。
- 需要测试时，编写兼容 `pytest` 的测试。
- 配置应显式、集中、可追踪。

## 不应

- 使用通配符导入。
- 使用可变全局状态。
- 使用裸 `except`。
- 使用未命名的魔法数字。
- 重复实现已有逻辑。

# 5. C++ 规范

## 必须

- 遵循 RAII。
- 保持 const correctness。
- 独占所有权使用 `std::unique_ptr`。
- 共享所有权仅在确实需要时使用 `std::shared_ptr`。
- 优先使用 `enum class`，避免无作用域枚举。
- 避免使用裸指针表达所有权。

## 应当

- 编译期常量优先使用 `constexpr`。
- 非拥有字符串参数优先使用 `std::string_view`。
- 可选值优先使用 `std::optional`。
- 可用时，非拥有连续区间优先使用 `std::span`。
- 有意识地使用移动语义。
- 仅在语义正确时标记 `noexcept`。

## 不应

- 使用裸 `new` 或 `delete`。
- 在头文件中使用 `using namespace std`。
- 在可使用 `constexpr` 时使用宏常量。
- 在有更安全替代方案时使用 C 风格数组。
- 混淆所有权和借用语义。

# 6. 验证

## 必须

- 在可行范围内验证修改。
- 优先使用项目提供的验证命令。
- 无法验证时必须说明原因。
- 验证失败时报告失败命令和关键原因。
- 不得声称执行过未实际执行的验证。

## 应当

适用时按以下顺序验证：

1. 测试
2. 构建
3. Lint 或静态检查
4. 运行时验证

## 禁止

- 编造测试结果。
- 忽略失败的验证。
- 用无关命令的成功结果证明修改正确。

---

# 7. Git 工作流

## 必须

- 生成提交信息前先查看 staged diff。
- 提交信息只能基于 staged diff 生成。
- 用户要求提交信息时，使用 Conventional Commits。
- 提交标题应简洁、具体。

## 应当

- 除非仓库要求英文，否则提交标题使用中文。
- 一个提交只包含一个逻辑修改。
- 必要时在提交正文中说明重要验证结果。

## 禁止

- 在提交信息中包含密钥、Token、密码或其它敏感信息。
- 使用“更新代码”“修复问题”等空泛标题。
- 描述 staged diff 中不存在的改动。

---

# 8. 工作区文件

## 必须

- 默认不创建工作区文档。
- 不把 `.codex` 当作永久知识库。
- 长期项目知识应写入仓库文档，而不是临时工作区文件。

## 可以

仅在以下情况创建工作区文件：

- 用户明确要求；
- 任务周期较长；
- 多个 Agent 需要交接上下文。

允许的工作区文件：

- `.codex/task.md`
- `.codex/context.md`
- `.codex/verification.md`

## 必须

使用工作区文件时：

- 保持内容简洁；
- 覆盖更新，而不是无限追加；
- 只保存摘要，不保存完整日志；
- 清理过期临时内容。

---

# 9. 回复格式

## 必须

最终回复应包含：

- 已完成内容；
- 验证情况；
- 存在风险或限制时明确说明。

## 应当

- 回复保持简洁。
- 有帮助时说明改动文件。
- 仅在下一步可执行时提供建议。

## 禁止

- 对常规修改做过度解释。
- 隐藏未完成或未验证的工作。
- 声称会在后台或未来继续执行任务。
